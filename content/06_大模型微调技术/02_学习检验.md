<a id="问题1"></a>
### 问题1：什么是参数高效微调（PEFT）？与全量微调相比有什么优势？

**参考答案**：

**PEFT定义**：
```
Parameter-Efficient Fine-Tuning
只微调少量参数（通常<1%），冻结大部分预训练参数
达到接近全量微调的效果
```

**主要方法分类**：

| 类型 | 代表方法 | 原理 |
|-----|---------|------|
| Adapter类 | LoRA, Adapter | 添加可学习的小模块 |
| Prompt类 | Prefix Tuning, P-Tuning | 添加可学习的提示向量 |
| 其他 | BitFit, IA³ | 只训练特定参数 |

**PEFT优势**：

1. **显存效率**
   - 全量微调7B模型需要~120GB
   - LoRA只需~16GB
   - QLoRA只需~6GB

2. **存储效率**
   - 全量微调：每个任务存一个完整模型
   - PEFT：每个任务只存小adapter（几十MB）

3. **训练速度**
   - 更新参数少，梯度计算快
   - 通信开销小

4. **减少灾难性遗忘**
   - 原始参数冻结
   - 保留预训练知识

<a id="问题2"></a>
### 问题2：请详细解释LoRA的原理，为什么它是有效的？

**参考答案**：

**核心假设**：
```
微调时的权重变化矩阵ΔW是低秩的
即ΔW可以分解为两个小矩阵的乘积：ΔW = BA

其中：
- W₀ ∈ ℝᵈˣᵈ（原始权重，冻结）
- B ∈ ℝᵈˣʳ（可训练）
- A ∈ ℝʳˣᵈ（可训练）
- r << d（低秩，通常r=8~64）
```

**前向计算**：
```
原始：h = W₀x
LoRA：h = W₀x + (α/r)BAx

α是缩放因子，用于调节LoRA的影响强度
```

**有效性原因**：

1. **低秩假设合理**
   - 研究表明微调的参数变化确实是低秩的
   - 下游任务只需要在预训练基础上小幅调整

2. **参数效率**
   ```
   原始参数：d × d
   LoRA参数：d × r + r × d = 2dr

   当d=4096, r=8时：
   压缩比 = d²/(2dr) = d/(2r) = 256倍
   ```

3. **初始化策略**
   - A初始化为随机高斯
   - B初始化为0
   - 初始时ΔW=0，从预训练状态开始

<a id="问题3"></a>
### 问题3：LoRA中的秩r应该如何选择？有什么影响？

**参考答案**：

**r的影响**：

| r值 | 参数量 | 表达能力 | 过拟合风险 | 显存占用 |
|----|-------|---------|-----------|---------|
| 小（4-8） | 很少 | 有限 | 低 | 低 |
| 中（16-32） | 适中 | 较强 | 中 | 中 |
| 大（64-128） | 较多 | 很强 | 高 | 较高 |

**选择建议**：

```
1. 简单任务（分类、简单问答）：r=8
2. 中等任务（翻译、摘要）：r=16-32
3. 复杂任务（代码生成、复杂推理）：r=32-64
4. 数据量少时：使用小r，避免过拟合
5. 数据量大时：可以尝试大r
```

**经验公式**：
```
通常alpha = 2r 或 alpha = r
实际影响 = alpha/r × LoRA输出

当alpha=2r时，相当于放大LoRA影响
当alpha=r时，影响系数为1
```

**实验建议**：
```python
# 从小r开始尝试
for r in [8, 16, 32, 64]:
    # 训练并评估
    # 选择效果和效率的平衡点
```

<a id="问题4"></a>
### 问题4：QLoRA相比LoRA做了哪些改进？为什么能大幅降低显存？

**参考答案**：

**QLoRA的三大创新**：

**1. 4-bit NormalFloat量化（NF4）**
```
将基础模型权重量化到4-bit
专门为正态分布的神经网络权重设计
比普通INT4精度更高
```

**2. 双重量化（Double Quantization）**
```
不仅量化权重，还量化量化参数
传统：每64个权重需要1个FP32量化常数
双重量化：量化常数也被量化到8-bit
额外节省0.5GB/B参数
```

**3. 分页优化器（Paged Optimizers）**
```
利用CPU内存处理显存峰值
当显存不足时，自动将优化器状态转移到CPU
避免OOM
```

**显存对比**：

| 模型 | 全量微调 | LoRA(FP16) | QLoRA(NF4) |
|-----|---------|------------|------------|
| 7B | 120GB | 50GB | 6GB |
| 13B | 200GB | 80GB | 10GB |
| 70B | 1000GB+ | 400GB | 48GB |

**降低显存的原因**：
```
1. 基础模型从FP16(2字节)→NF4(0.5字节)：4倍压缩
2. 梯度只计算LoRA部分（<1%参数）
3. 优化器状态也只针对LoRA参数
4. 双重量化额外节省空间
```

<a id="问题5"></a>
### 问题5：什么是NF4量化？为什么QLoRA选择这种量化方式？

**参考答案**：

**NF4（NormalFloat4）原理**：

```
神经网络权重通常服从正态分布N(0, σ²)
NF4的量化点选择：让每个量化区间包含相同概率的权重

传统INT4：均匀分布量化点
NF4：根据正态分布CDF分布量化点

量化点 = quantile(N(0,1), [0/16, 1/16, ..., 15/16])
```

**NF4 vs INT4对比**：

| 特性 | INT4 | NF4 |
|-----|------|-----|
| 量化点分布 | 均匀 | 根据正态分布 |
| 对神经网络权重 | 不是最优 | 更优 |
| 量化误差 | 较大 | 较小 |
| 信息保留 | 一般 | 更好 |

**为什么选择NF4**：

1. **理论最优**
   - 当数据服从正态分布时，NF4是信息论意义上最优的4-bit量化

2. **实践效果**
   - 相同bit数下，NF4比INT4精度损失更小
   - QLoRA论文实验：NF4 > FP4 > INT4

3. **计算兼容**
   - 可以高效地反量化为FP16/BF16计算

<a id="问题6"></a>
### 问题6：Adapter Tuning和LoRA有什么区别？各自的优缺点？

**参考答案**：

**结构对比**：

```
Adapter Tuning：
┌─────────┐
│ 原始层   │（冻结）
└────┬────┘
     ↓
┌─────────┐
│ Adapter │（可训练，串行）
└────┬────┘
     ↓

LoRA：
┌─────────┐     ┌─────┐
│ 原始层   │  +  │B × A│（并行旁路）
└────┬────┘     └──┬──┘
     │             │
     └──────┬──────┘
            ↓
```

**详细对比**：

| 特性 | Adapter | LoRA |
|-----|---------|------|
| 连接方式 | 串行 | 并行 |
| 推理延迟 | 有额外计算 | 可合并消除 |
| 参数效率 | 较低 | 更高 |
| 实现复杂度 | 中等 | 简单 |
| 多任务支持 | 需切换模块 | 可直接融合 |

**Adapter优势**：
- 理论表达能力更强（有非线性激活）
- 更好的任务隔离

**LoRA优势**：
- 推理可无损合并
- 参数更少
- 更易于实现和部署

**结论**：大多数场景推荐LoRA

<a id="问题7"></a>
### 问题7：Prefix Tuning和P-Tuning有什么区别？

**参考答案**：

**Prefix Tuning**：
```
在每一层的Attention中添加可学习的prefix向量
添加到Key和Value：
K' = [K_prefix; K_input]
V' = [V_prefix; V_input]

特点：
- 每层都有独立的prefix
- 不修改输入embedding
- 参数量：prefix_len × num_layers × hidden_size × 2
```

**P-Tuning v1**：
```
只在输入层添加可学习的连续向量
用MLP生成这些向量（增加表达能力）

特点：
- 只在输入层
- 使用LSTM/MLP生成prompt
- 参数量较少
```

**P-Tuning v2**：
```
在每一层都添加可学习的prompt向量
实际上等价于Prefix Tuning

特点：
- 深层prompt
- 效果接近fine-tuning
```

**对比表格**：

| 方法 | Prompt位置 | 参数量 | 效果 | 适用模型 |
|-----|-----------|-------|------|---------|
| Prompt Tuning | 仅输入层 | 最少 | 较弱 | 大模型(10B+) |
| P-Tuning v1 | 输入层+MLP | 少 | 中等 | 中小模型 |
| P-Tuning v2 | 每层 | 较多 | 好 | 各种规模 |
| Prefix Tuning | 每层K,V | 较多 | 好 | 各种规模 |

<a id="问题8"></a>
### 问题8：什么是指令微调（Instruction Tuning）？为什么重要？

**参考答案**：

**定义**：
```
指令微调是让模型学会理解和遵循自然语言指令的训练方法
通过大量（指令，输入，输出）三元组进行训练
```

**数据格式示例**：
```json
{
  "instruction": "请将以下句子翻译成英文",
  "input": "今天天气很好",
  "output": "The weather is nice today."
}
```

**重要性**：

| 方面 | 说明 |
|-----|------|
| 提升可用性 | 用户可以用自然语言与模型交互 |
| 泛化能力 | 学会遵循指令后，可泛化到新任务 |
| 减少prompt工程 | 不需要精心设计few-shot示例 |
| 对齐用户意图 | 模型输出更符合期望 |

**关键发现**：
```
1. 任务多样性很重要
   - FLAN使用1836个任务
   - 多样性比数量更重要

2. 指令格式要一致
   - 统一的模板有助于学习

3. 高质量数据很关键
   - 小而精的数据集效果可能超过大而杂的
```

<a id="问题9"></a>
### 问题9：指令微调时，为什么只对回答部分计算Loss？

**参考答案**：

**原因分析**：

**1. 训练目标不同**
```
目标：让模型学会根据指令生成正确回答
不是让模型学会生成指令本身

如果对指令也计算Loss：
- 模型会花精力学习生成指令
- 这不是我们想要的能力
```

**2. 避免信息泄露**
```
输入：[指令][问题]
输出：[回答]

如果对全部计算Loss，模型可能学会：
- 根据[问题]预测[回答]的开头
- 而不是理解指令的含义
```

**3. 提高训练效率**
```
指令部分通常很长
回答部分是重点

只对回答计算Loss：
- 梯度信号更集中
- 训练更高效
```

**实现方式**：
```python
def create_labels(input_ids, instruction_length):
    labels = input_ids.clone()
    # 将指令部分设为-100，不计算Loss
    labels[:instruction_length] = -100
    return labels

# CrossEntropyLoss会忽略-100的位置
loss = F.cross_entropy(logits, labels, ignore_index=-100)
```

<a id="问题10"></a>
### 问题10：如何选择合适的微调方法？请给出决策依据。

**参考答案**：

**决策流程**：

```
Step 1: 评估资源
├── GPU显存充足（>100GB）→ 考虑全量微调
├── GPU显存中等（20-80GB）→ LoRA
└── GPU显存有限（<16GB）→ QLoRA

Step 2: 评估需求
├── 需要最佳效果 → 全量微调/LoRA+大r
├── 需要快速迭代 → LoRA/QLoRA
└── 多任务部署 → PEFT方法

Step 3: 评估部署
├── 推理延迟敏感 → LoRA（可合并）
├── 多任务切换 → Adapter/LoRA（保持base）
└── 边缘部署 → QLoRA+量化推理
```

**详细对比表**：

| 场景 | 推荐方法 | 理由 |
|-----|---------|------|
| 资源充足，追求极致效果 | 全量微调 | 效果最好 |
| 通用场景 | LoRA | 平衡效果和效率 |
| 消费级GPU | QLoRA | 显存需求最低 |
| 快速原型验证 | Prompt Tuning | 参数最少，最快 |
| 多任务服务 | LoRA | 多个adapter可动态切换 |
| 边缘设备部署 | QLoRA+INT4推理 | 全链路量化 |

<a id="问题11"></a>
### 问题11：LoRA的target_modules应该如何选择？

**参考答案**：

**常见选择**：

```
Transformer Block:
├── Attention
│   ├── q_proj  ✓ 推荐
│   ├── k_proj  △ 可选
│   ├── v_proj  ✓ 推荐
│   └── o_proj  △ 可选
└── FFN
    ├── gate_proj  △ 可选
    ├── up_proj    △ 可选
    └── down_proj  △ 可选
```

**选择策略**：

| 策略 | 模块 | 参数量 | 效果 |
|-----|------|-------|------|
| 最小化 | q, v | 最少 | 好 |
| 平衡 | q, k, v, o | 中等 | 更好 |
| 最大化 | 全部Attention+FFN | 最多 | 最好但可能过拟合 |

**实验发现**：
```
1. 只用q_proj和v_proj效果就很好
   - 这是LoRA原论文的推荐

2. 加入FFN可以提升复杂任务
   - 特别是代码生成、数学推理

3. 不同模型可能有差异
   - LLaMA系列：q, v, k, o效果好
   - GPT系列：q, v足够
```

**代码示例**：
```python
# 最小配置
target_modules = ["q_proj", "v_proj"]

# 平衡配置
target_modules = ["q_proj", "k_proj", "v_proj", "o_proj"]

# 完整配置（LLaMA）
target_modules = [
    "q_proj", "k_proj", "v_proj", "o_proj",
    "gate_proj", "up_proj", "down_proj"
]
```

<a id="问题12"></a>
### 问题12：微调时遇到过拟合怎么办？

**参考答案**：

**过拟合表现**：
```
- 训练Loss持续下降
- 验证Loss开始上升
- 生成内容重复、死板
- 只会模仿训练数据格式
```

**解决方案**：

| 方法 | 说明 | 操作 |
|-----|------|------|
| 减小r | 降低模型容量 | r从64减到16或8 |
| 增加dropout | 正则化 | lora_dropout从0.05增到0.1 |
| 早停 | 在验证Loss最低时停止 | 设置early_stopping |
| 数据增强 | 增加数据多样性 | 同义改写、回译 |
| 减少epoch | 防止过度训练 | 通常1-3个epoch足够 |
| 增大数据量 | 根本解决 | 收集更多数据 |

**代码示例**：
```python
# 增加dropout
lora_config = LoraConfig(
    r=8,  # 减小r
    lora_dropout=0.1,  # 增加dropout
    ...
)

# 训练时使用早停
training_args = TrainingArguments(
    ...
    evaluation_strategy="steps",
    eval_steps=100,
    load_best_model_at_end=True,
    metric_for_best_model="eval_loss",
)
```

<a id="问题13"></a>
### 问题13：LoRA训练后如何合并权重？有什么好处？

**参考答案**：

**合并原理**：
```
训练时：h = W₀x + BAx
合并后：h = (W₀ + BA)x = W_merged × x

W_merged = W₀ + (α/r) × B @ A
```

**合并代码**：
```python
# 使用PEFT库合并
merged_model = model.merge_and_unload()

# 保存合并后的模型
merged_model.save_pretrained("merged_model")

# 手动合并（了解原理）
for name, module in model.named_modules():
    if hasattr(module, 'lora_A'):
        W = module.weight.data
        A = module.lora_A.weight.data
        B = module.lora_B.weight.data
        scaling = module.scaling
        W += scaling * (B @ A)
```

**合并的好处**：

| 好处 | 说明 |
|-----|------|
| 消除推理延迟 | 不再需要额外的矩阵乘法 |
| 简化部署 | 只需要一个模型文件 |
| 兼容现有框架 | 可以用标准推理流程 |
| 减少内存 | 不需要存储LoRA参数 |

**不合并的场景**：
```
1. 多任务部署
   - 需要动态切换不同LoRA
   - 基础模型共享

2. 增量训练
   - 后续可能继续微调

3. A/B测试
   - 需要比较不同LoRA效果
```

<a id="问题14"></a>
### 问题14：什么是灾难性遗忘？PEFT如何缓解这个问题？

**参考答案**：

**灾难性遗忘定义**：
```
模型在学习新任务时，遗忘之前学会的知识
全量微调时尤为严重

表现：
- 微调后模型通用能力下降
- 只会做微调任务，其他任务效果变差
- 语言生成能力退化
```

**原因**：
```
全量微调时：
- 所有参数都被更新
- 新任务的梯度覆盖旧知识
- 权重偏向新任务的分布
```

**PEFT如何缓解**：

| 机制 | 说明 |
|-----|------|
| 参数冻结 | 预训练参数不变，知识保留 |
| 小参数量 | 新增参数少，对原模型影响小 |
| 旁路结构 | LoRA是附加的，不修改原权重 |
| 可控影响 | alpha/r控制LoRA影响强度 |

**对比实验**：
```
全量微调后：
- 新任务：95%
- 原有能力：60%（严重下降）

LoRA微调后：
- 新任务：92%
- 原有能力：88%（轻微下降）
```

**进一步缓解方法**：
```python
# 1. 控制学习率
learning_rate = 2e-5  # 比全量微调更小

# 2. 减小alpha
lora_config = LoraConfig(
    r=8,
    lora_alpha=8,  # alpha=r，影响适中
)

# 3. 混合训练数据
# 加入部分通用数据保持能力
```

<a id="问题15"></a>
### 问题15：如何评估微调后模型的效果？

**参考答案**：

**评估维度**：

| 维度 | 评估方法 | 指标 |
|-----|---------|------|
| 任务效果 | 任务特定测试集 | 准确率/F1/BLEU等 |
| 通用能力 | 通用benchmark | MMLU/C-Eval等 |
| 生成质量 | 人工评估 | 流畅度/相关性/准确性 |
| 指令遵循 | 指令测试集 | 格式正确率 |

**常用评估方法**：

**1. 自动评估**
```python
# 使用lm-evaluation-harness
from lm_eval import evaluator

results = evaluator.simple_evaluate(
    model=model,
    tasks=["mmlu", "hellaswag", "arc_easy"],
    batch_size=8,
)
```

**2. 人工评估**
```
设计评估问题：
- 领域相关问题
- 格式要求问题
- 边界情况

评估标准：
- 正确性（1-5分）
- 完整性（1-5分）
- 格式遵循（是/否）
```

**3. A/B测试**
```
对比：
- 微调前 vs 微调后
- 不同微调配置
- 不同数据量

方法：
- 盲评（评估者不知道来源）
- 胜率统计
```

**评估最佳实践**：
```
1. 准备holdout测试集
   - 不参与训练
   - 代表真实使用场景

2. 多维度评估
   - 不只看训练Loss
   - 检查通用能力是否保留

3. 定性检查
   - 人工查看生成样例
   - 发现自动指标遗漏的问题
```
