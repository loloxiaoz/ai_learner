### 问题1：大模型推理的主要瓶颈是什么？Prefill和Decode有什么区别？

**参考答案**：

**推理两阶段**：

| 阶段 | Prefill | Decode |
|-----|---------|--------|
| 处理内容 | 输入prompt | 逐token生成 |
| 特点 | 可并行处理 | 串行生成 |
| 瓶颈 | 计算密集 | 内存带宽密集 |
| GPU利用率 | 高 | 低 |

**瓶颈分析**：

```
Prefill阶段：
- 一次处理所有输入token
- 大量矩阵乘法，计算密集
- GPU利用率高，计算是瓶颈

Decode阶段：
- 每次只生成1个token
- 每个token都要读取全部模型参数
- 参数量远大于计算量
- 内存带宽是瓶颈

计算强度（Arithmetic Intensity）：
AI = FLOPs / Bytes

Decode时AI很低，内存带宽无法喂饱GPU
```

**优化方向**：

```
Prefill优化：
- Flash Attention减少内存
- 更高效的并行

Decode优化：
- 量化减少参数大小
- KV Cache减少重复计算
- 批处理提高利用率
```

<a id="问题2"></a>
### 问题2：什么是模型量化？有哪些常见的量化方法？

**参考答案**：

**量化定义**：
```
将高精度浮点数（FP32/FP16）转换为低精度（INT8/INT4）
目的：减少内存占用，加速推理
```

**量化公式**：
```
量化：x_int = round(x_float / scale) + zero_point
反量化：x_float = (x_int - zero_point) * scale
```

**常见量化方法**：

| 方法 | 类型 | 特点 |
|-----|------|------|
| 动态量化 | PTQ | 运行时确定scale |
| 静态量化 | PTQ | 校准数据确定scale |
| QAT | 训练时量化 | 最高精度 |
| GPTQ | PTQ | 二阶信息优化 |
| AWQ | PTQ | 激活感知 |

**按位数分类**：

```
INT8量化：
- 内存减少2×（FP16→INT8）
- 精度损失很小
- 普遍支持

INT4量化：
- 内存减少4×（FP16→INT4）
- 精度损失较小
- GPTQ/AWQ主流

更低位数（INT3/INT2）：
- 研究阶段
- 精度损失较大
```

<a id="问题3"></a>
### 问题3：GPTQ和AWQ有什么区别？各自的原理是什么？

**参考答案**：

**GPTQ原理**：
```
GPT Quantization
基于二阶（Hessian）信息的后训练量化

核心思想：
1. 逐列量化权重矩阵
2. 用Hessian矩阵指导量化顺序
3. 量化误差补偿到未量化的列

算法：
for i in columns:
    q_i = quantize(W[:, i])
    error = W[:, i] - q_i
    W[:, i+1:] -= error * H[i, i+1:] / H[i, i]

优点：量化误差最小化
缺点：需要较多校准数据，量化较慢
```

**AWQ原理**：
```
Activation-aware Weight Quantization
激活感知的权重量化

核心发现：
- 少量权重对输出影响很大（显著权重）
- 保护这些权重可以大幅减少误差

方法：
1. 通过校准数据找出激活值大的通道
2. 对应权重乘以大scale
3. 量化时这些权重精度更高

优点：精度更好，量化更快
缺点：需要定制kernel
```

**对比**：

| 特性 | GPTQ | AWQ |
|-----|------|-----|
| 原理 | 误差补偿 | 显著权重保护 |
| 精度 | 很好 | 更好 |
| 速度 | 较慢 | 较快 |
| 推理速度 | 快 | 更快 |
| 实现复杂度 | 中 | 中 |

<a id="问题4"></a>
### 问题4：什么是KV Cache？为什么需要它？

**参考答案**：

**KV Cache定义**：
```
在自回归生成时，缓存之前token的Key和Value
避免重复计算，用空间换时间
```

**为什么需要**：

```
不用KV Cache：
Step 1: 计算 Attention(Q₁, K₁, V₁)
Step 2: 计算 Attention(Q₂, [K₁,K₂], [V₁,V₂])  // 重复计算K₁,V₁
Step 3: 计算 Attention(Q₃, [K₁,K₂,K₃], [V₁,V₂,V₃])  // 重复更多

复杂度：O(n³)

用KV Cache：
Step 1: 计算K₁,V₁，缓存
Step 2: 计算K₂,V₂，拼接缓存[K₁,K₂], [V₁,V₂]
Step 3: 计算K₃,V₃，拼接缓存

复杂度：O(n²)
```

**内存计算**：
```
KV Cache大小 = 2 × layers × batch × seq_len × hidden × precision

LLaMA-7B, FP16, batch=1, seq=2048:
= 2 × 32 × 1 × 2048 × 4096 × 2 = 1GB

问题：长上下文+大batch时显存爆炸
```

**优化方法**：
```
1. GQA：减少KV头数
2. KV Cache量化
3. PagedAttention
4. Sliding Window
```

<a id="问题5"></a>
### 问题5：GQA相比MHA有什么优势？原理是什么？

**参考答案**：

**三种Attention对比**：

```
MHA (Multi-Head Attention)：
- 每个头独立的Q, K, V
- Q头数 = K头数 = V头数 = n_heads
- KV Cache大小 ∝ n_heads

MQA (Multi-Query Attention)：
- 所有头共享一组K, V
- Q头数 = n_heads, K头数 = V头数 = 1
- KV Cache大小 ∝ 1
- 压缩n_heads倍，但精度损失大

GQA (Grouped-Query Attention)：
- 每组头共享一组K, V
- Q头数 = n_heads, K头数 = V头数 = n_groups
- KV Cache大小 ∝ n_groups
- 折中方案
```

**GQA优势**：

| 方面 | MHA | GQA | MQA |
|-----|-----|-----|-----|
| KV Cache | 1× | 1/4 ~ 1/8 | 1/n_heads |
| 精度 | 基准 | 接近MHA | 有损失 |
| 推理速度 | 基准 | 更快 | 最快 |

**实际应用**：
```
LLaMA-2-70B使用GQA：
- n_heads = 64
- n_kv_heads = 8
- 每8个Q头共享1组KV
- KV Cache减少8×
```

<a id="问题6"></a>
### 问题6：Flash Attention是如何优化内存和速度的？

**参考答案**：

**问题背景**：
```
标准Attention：
S = QK^T          // N×N矩阵
P = softmax(S)    // N×N矩阵
O = PV            // N×d矩阵

问题：
- S和P是N×N矩阵，内存O(N²)
- 长序列时内存爆炸
- 需要在HBM和SRAM间频繁传输
```

**Flash Attention原理**：

```
核心思想：利用GPU内存层级

GPU内存层级：
- SRAM：~20MB，19TB/s（快但小）
- HBM：~40GB，1.5TB/s（慢但大）

优化策略：
1. 分块计算（Tiling）
   - 将Q,K,V分成小块
   - 每块在SRAM中计算
   - 不存储完整的N×N矩阵

2. 重计算（Recomputation）
   - 反向传播时不存储S和P
   - 需要时重新计算
   - 用计算换内存

3. 在线Softmax
   - 分块计算时动态更新softmax
   - 数学上等价于全局softmax
```

**效果**：
```
内存：O(N²) → O(N)
速度：2-4×加速（减少HBM访问）
精度：完全无损
```

<a id="问题7"></a>
### 问题7：什么是PagedAttention？vLLM是如何实现的？

**参考答案**：

**传统KV Cache问题**：
```
预分配方式：
- 为每个请求预分配max_seq_len的KV Cache
- 实际使用可能很少
- 内存利用率低（~30%）
- 内存碎片化
```

**PagedAttention原理**：

```
类似操作系统的虚拟内存管理

核心思想：
1. 将KV Cache分成固定大小的页（block）
2. 每个block存储固定数量token的KV
3. 动态分配页，按需扩展
4. 请求完成释放页，可被复用

实现：
- Block Table：记录每个序列的块映射
- Physical Blocks：实际存储KV的内存块
- 逻辑地址→物理地址映射
```

**vLLM实现**：

```python
# 逻辑视图
sequence_1: [block_0, block_1, block_2]
sequence_2: [block_3, block_4]

# 物理存储
physical_blocks = {
    0: KV_data_0,  # 属于seq_1
    1: KV_data_1,  # 属于seq_1
    2: KV_data_2,  # 属于seq_1
    3: KV_data_3,  # 属于seq_2
    4: KV_data_4,  # 属于seq_2
    5: None,       # 空闲
    ...
}

# seq_1完成后，block 0,1,2可被复用
```

**效果**：
```
内存利用率：30% → 90%+
支持更大batch size
减少OOM
```

<a id="问题8"></a>
### 问题8：什么是Continuous Batching？有什么好处？

**参考答案**：

**传统Static Batching问题**：

```
Static Batching：
- 等待凑齐一个batch
- batch内所有请求一起处理
- 等最长的请求完成才能处理下一批

问题：
请求1：[####]
请求2：[########]
请求3：[################]
所有请求等request_3完成
短请求等待时间长，利用率低
```

**Continuous Batching**：

```
动态批处理：
- 请求完成立即移出
- 新请求立即加入
- 无需等待

时间线：
t=0: [req1, req2, req3] 开始
t=1: [req1完成, req2, req3, req4加入]
t=2: [req2完成, req3, req4, req5加入]
...

效果：
- 短请求快速返回
- GPU持续被利用
- 吞吐量大幅提升
```

**好处**：

| 方面 | Static | Continuous |
|-----|--------|-----------|
| 短请求延迟 | 高 | 低 |
| GPU利用率 | 低 | 高 |
| 吞吐量 | 低 | 高 |
| 实现复杂度 | 简单 | 复杂 |

<a id="问题9"></a>
### 问题9：什么是Speculative Decoding？原理是什么？

**参考答案**：

**定义**：
```
推测解码：用小模型快速生成草稿，大模型验证
无损加速技术
```

**原理**：

```
流程：
1. 小模型（draft）快速生成K个token
2. 大模型（target）一次前向验证所有K个token
3. 如果draft的token与target一致，接受
4. 从第一个不一致处继续

示例：
Draft生成：[A, B, C, D, E]
Target验证：[A, B, X, -, -]
接受：[A, B]
从C的位置用target的输出X继续

加速原理：
- 小模型生成很快
- 大模型一次验证K个token
- 相当于大模型一次前向产生多个token
```

**实现**：

```python
def speculative_decode(draft_model, target_model, prompt, K=5):
    tokens = prompt

    while not finished:
        # 1. Draft模型生成K个token
        draft_tokens = draft_model.generate(tokens, K)

        # 2. Target模型验证
        # 一次前向处理所有候选token
        target_logits = target_model(tokens + draft_tokens)

        # 3. 比较并接受匹配的token
        accepted = 0
        for i in range(K):
            if matches(draft_tokens[i], target_logits[i]):
                accepted += 1
            else:
                break

        tokens += draft_tokens[:accepted]
        if accepted < K:
            # 用target的输出继续
            tokens += sample(target_logits[accepted])
```

**加速效果**：
```
加速比取决于接受率（acceptance rate）
典型：1.5-3×加速
任务相关：简单任务接受率高，加速明显
```

**参考答案**：

**对比**：

| 指标 | FP16 | INT8 | INT4 |
|-----|------|------|------|
| 内存 | 2B/param | 1B/param | 0.5B/param |
| 精度 | 100% | ~99% | ~97% |
| 推理速度 | 基准 | 1.5-2× | 2-3× |
| 支持度 | 全部 | 广泛 | GPTQ/AWQ |

**选择依据**：

```
选择INT8：
- 精度要求高
- 对量化敏感的任务（数学、代码）
- 简单部署，兼容性好

选择INT4：
- 资源受限（消费级GPU）
- 对精度容忍度高
- 追求性价比

选择FP16：
- 不能接受任何精度损失
- 资源充足
```

<a id="问题11"></a>
### 问题11：部署7B/13B/70B模型分别需要什么配置？

**参考答案**：

**显存估算公式**：
```
显存 ≈ 参数量 × 精度字节 × 1.2（overhead）

7B FP16: 7 × 10⁹ × 2 × 1.2 ≈ 17GB
7B INT4: 7 × 10⁹ × 0.5 × 1.2 ≈ 4.2GB
```

**详细配置**：

| 模型 | 精度 | 显存需求 | 推荐GPU |
|-----|------|---------|---------|
| 7B | FP16 | 17GB | A100 40GB / A10 24GB |
| 7B | INT8 | 10GB | RTX 4090 24GB |
| 7B | INT4 | 5GB | RTX 3060 12GB |
| 13B | FP16 | 30GB | A100 40GB |
| 13B | INT8 | 16GB | RTX 4090 24GB |
| 13B | INT4 | 8GB | RTX 3080 10GB |
| 70B | FP16 | 140GB | 2×A100 80GB |
| 70B | INT8 | 75GB | 1×A100 80GB |
| 70B | INT4 | 38GB | 1×A100 40GB |

**注意事项**：
```
1. KV Cache额外占用
   - 长上下文需要更多显存
   - batch size影响

2. 峰值显存
   - 实际需求比理论值高
   - 预留20%余量

3. 吞吐量考虑
   - batch size影响吞吐
   - 需要额外显存支持batch
```

### 问题12：如何评估推理性能？关键指标有哪些？

**参考答案**：

**关键指标**：

| 指标 | 定义 | 重要性 |
|-----|------|-------|
| TTFT | Time To First Token | 用户体验 |
| TPS | Tokens Per Second | 吞吐量 |
| Latency | 单请求延迟 | 响应速度 |
| Throughput | 每秒请求数 | 系统容量 |
| GPU Util | GPU利用率 | 资源效率 |

**测量方法**：

```python
import time

def benchmark_inference(model, prompts, max_tokens=100):
    results = []

    for prompt in prompts:
        start = time.time()

        # 首token时间
        first_token_time = None
        tokens = []

        for token in model.generate_stream(prompt, max_tokens):
            if first_token_time is None:
                first_token_time = time.time() - start
            tokens.append(token)

        total_time = time.time() - start

        results.append({
            "ttft": first_token_time,
            "total_time": total_time,
            "tokens": len(tokens),
            "tps": len(tokens) / (total_time - first_token_time),
        })

    return {
        "avg_ttft": mean([r["ttft"] for r in results]),
        "avg_tps": mean([r["tps"] for r in results]),
        "avg_latency": mean([r["total_time"] for r in results]),
    }
```

**性能基准**：
```
好的性能参考（7B模型，单A100）：
- TTFT: < 100ms
- TPS: > 50 tokens/s
- GPU利用率: > 70%

优化目标：
- 降低TTFT：优化Prefill
- 提高TPS：优化Decode
- 提高吞吐：增大batch
```

### 问题14：长上下文场景下如何优化推理？

**参考答案**：

**长上下文挑战**：
```
1. KV Cache显存爆炸
   128K上下文 → KV Cache 10GB+/layer

2. Attention计算量
   O(n²)复杂度，128K比4K慢1000倍

3. 精度问题
   长距离依赖可能丢失
```

**优化方法**：

| 方法 | 原理 | 效果 |
|-----|------|------|
| GQA | 减少KV头数 | KV Cache减少4-8× |
| KV Cache量化 | INT8/FP8存储 | 显存减半 |
| Sliding Window | 只保留窗口内KV | 显存O(window) |
| 稀疏Attention | 只计算部分注意力 | 计算量减少 |
| Ring Attention | 分布式处理长序列 | 扩展上下文 |

**实践建议**：

```python
# 长上下文配置
config = {
```

**参考答案**：

**高可用架构**：

**关键措施**：

| 方面 | 措施 |
|-----|------|
| 冗余 | 多实例部署，至少2副本 |
| 负载均衡 | 请求分发，避免单点过载 |
| 健康检查 | 定期检测服务状态 |
| 故障转移 | 自动切换到健康节点 |
| 限流 | 保护服务不被打垮 |
| 监控告警 | 及时发现问题 |

**Kubernetes部署示例**：

```yaml
apiVersion: apps/v1

  name: vllm-service

  selector:
    matchLabels:
      app: vllm
  template:
    spec:
      containers:
      - name: vllm
        image: vllm/vllm-openai:latest
        resources:
          limits:
            nvidia.com/gpu: 1
        livenessProbe:  # 健康检查
          httpGet:
            path: /health
            port: 8000
          periodSeconds: 10
        readinessProbe:
          httpGet:
            path: /ready
            port: 8000
---
apiVersion: v1

  name: vllm-service
spec:
  type: LoadBalancer
  ports:
  - port: 80
    targetPort: 8000
  selector:
    app: vllm
```
