<a id="问题1"></a>
### 问题1：大模型训练为什么需要分布式？单卡训练有什么限制？

**参考答案**：

**单卡限制**：

| 限制类型 | 说明 |
|---------|------|
| **内存限制** | 模型参数+优化器状态+梯度+激活值需要大量显存 |
| **计算限制** | 单卡算力有限，训练时间过长 |
| **通信限制** | 无法利用多卡并行 |

**内存计算示例（175B模型）**：
```
参数（FP16）：175B × 2字节 = 350GB
优化器状态（Adam FP32）：175B × 12字节 = 2100GB
梯度（FP16）：175B × 2字节 = 350GB
激活值：取决于batch size

总计：约2800GB，远超单卡80GB
```

**分布式训练的必要性**：
1. 模型参数无法装入单卡
2. 训练时间需要缩短（从年到天）
3. 增大batch size提升训练稳定性

<a id="问题2"></a>
### 问题2：请解释数据并行、张量并行、流水线并行的原理

**参考答案**：

**数据并行（Data Parallelism）**：
```
原理：每个GPU持有完整模型，处理不同数据
流程：
1. 数据分成N份（N=GPU数）
2. 每个GPU独立前向反向
3. AllReduce同步梯度
4. 各自更新参数

优点：实现简单，扩展性好
缺点：模型必须装入单卡
```

**张量并行（Tensor Parallelism）**：
```
原理：将单层的权重切分到多个GPU
示例（线性层列切分）：
Y = X[W₁|W₂] = [XW₁|XW₂]
GPU 0计算XW₁，GPU 1计算XW₂

优点：可以训练超大单层
缺点：通信频繁，需要高带宽
```

**流水线并行（Pipeline Parallelism）**：
```
原理：将模型按层切分，不同GPU负责不同层
示例（24层模型，4 GPU）：
GPU 0: Layer 0-5
GPU 1: Layer 6-11
GPU 2: Layer 12-17
GPU 3: Layer 18-23

优点：通信量小
缺点：存在Pipeline Bubble
```

<a id="问题3"></a>
### 问题3：什么是ZeRO优化？ZeRO-1/2/3有什么区别？

**参考答案**：

**ZeRO原理**：Zero Redundancy Optimizer，通过分片消除冗余存储

**三个阶段**：

| 阶段 | 分片内容 | 内存节省 | 通信量 |
|-----|---------|---------|-------|
| ZeRO-1 | 优化器状态 | 4× | 基本不变 |
| ZeRO-2 | + 梯度 | 8× | 基本不变 |
| ZeRO-3 | + 参数 | N× | 增加50% |

**ZeRO-1**：
```
优化器状态（Adam的m和v）分片存储
每个GPU只存储1/N的优化器状态
参数和梯度仍完整存储
```

**ZeRO-2**：
```
在ZeRO-1基础上，梯度也分片
反向传播时，梯度按需Reduce到对应GPU
```

**ZeRO-3**：
```
参数也分片存储
前向时：AllGather收集需要的参数
反向时：AllGather参数+ReduceScatter梯度
最激进的内存优化
```

**选择建议**：
- 模型能装入单卡：ZeRO-1/2
- 模型装不下：ZeRO-3
- 要极致性能：结合张量并行+ZeRO-1

<a id="问题4"></a>
### 问题4：为什么大模型训练使用BF16而不是FP16？

**参考答案**：

**FP16 vs BF16**：

| 特性 | FP16 | BF16 |
|-----|------|------|
| 位数 | 16 | 16 |
| 指数位 | 5 | 8 |
| 尾数位 | 10 | 7 |
| 数值范围 | ±65504 | ±3.4×10³⁸ |
| 精度 | 较高 | 较低 |

**BF16优势**：

1. **数值范围与FP32相同**
   - 不容易溢出
   - 不需要Loss Scaling

2. **大模型训练更稳定**
   - FP16在大梯度时容易溢出
   - BF16的范围足够处理极端值

3. **硬件支持好**
   - A100/H100原生支持
   - 与FP32无缝转换

**什么时候用FP16**：
- 老GPU（不支持BF16）
- 需要更高精度
- 配合Loss Scaling使用

<a id="问题5"></a>
### 问题5：什么是混合精度训练？为什么需要Loss Scaling？

**参考答案**：

**混合精度训练**：
```
使用低精度（FP16/BF16）加速计算
保留FP32主权重保证精度

流程：
1. FP32权重 → 转换 → FP16权重
2. FP16前向传播
3. FP16反向传播
4. FP16梯度 → 转换 → FP32梯度
5. 用FP32梯度更新FP32权重
```

**Loss Scaling原因**：
```
FP16的表示范围：[6×10⁻⁸, 65504]
小梯度（如1×10⁻⁸）会下溢为0

Loss Scaling解决：
1. Loss × 缩放因子S（如1024）
2. 梯度也被放大S倍
3. 更新前梯度 ÷ S

小梯度被放大后进入FP16可表示范围
```

**动态Loss Scaling**：
```python
scaler = GradScaler()

for data, target in dataloader:
    optimizer.zero_grad()
    with autocast():
        output = model(data)
        loss = loss_fn(output, target)

    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()  # 动态调整scale factor
```

<a id="问题6"></a>
### 问题6：什么是梯度检查点？它是如何节省内存的？

**参考答案**：

**问题背景**：
```
反向传播需要前向传播的激活值
激活值内存 = 层数 × batch_size × seq_len × hidden_dim
大模型的激活值占用巨大
```

**梯度检查点原理**：
```
不保存所有激活值
只保存"检查点"处的激活值
反向传播时，从检查点重新计算中间激活值
```

**示意图**：
```
标准训练（保存所有激活）：
Layer 1 → [a₁] → Layer 2 → [a₂] → Layer 3 → [a₃] → Loss

检查点训练（只保存检查点）：
Layer 1 → [a₁✓] → Layer 2 → [丢弃] → Layer 3 → [a₃✓] → Loss

反向时重算：
从a₁重新计算a₂
```

**权衡**：
```
内存：O(L) → O(√L)，L=层数
计算：增加约33%（重新前向）

通常每隔几层设一个检查点
或每个Transformer block一个检查点
```

<a id="问题7"></a>
### 问题7：如何设计千亿参数模型的3D并行策略？

**参考答案**：

**设计原则**：

1. **张量并行**：节点内（NVLink高带宽）
2. **流水线并行**：跨节点（较低通信）
3. **数据并行**：剩余GPU

**示例设计（175B模型，512 A100）**：

```
硬件假设：
- 64个节点，每节点8张A100
- 节点内NVLink，节点间InfiniBand

并行策略：
- TP = 8（节点内8卡张量并行）
- PP = 8（8个流水线阶段）
- DP = 8（8路数据并行）

8 × 8 × 8 = 512 GPU

每个流水线阶段：
- 参数量：175B / 8 ≈ 22B
- 分到8张卡（张量并行）：22B / 8 ≈ 2.75B/卡
```

**配置要点**：

| 并行类型 | 考虑因素 |
|---------|---------|
| TP大小 | 受限于节点内GPU数，通常4-8 |
| PP大小 | 考虑Pipeline Bubble，通常4-16 |
| DP大小 | 剩余GPU数，越大batch越大 |

<a id="问题8"></a>
### 问题8：训练中遇到Loss Spike怎么处理？

**参考答案**：

**诊断步骤**：

1. **检查是否是数据问题**
   - 查看spike时的batch数据
   - 是否有异常样本

2. **检查梯度**
   - 梯度范数是否异常大
   - 哪些层的梯度异常

3. **检查学习率**
   - 是否在lr变化点
   - lr是否过大

**解决方案**：

| 方案 | 说明 |
|-----|------|
| 梯度裁剪 | `torch.nn.utils.clip_grad_norm_(params, max_norm=1.0)` |
| 减小学习率 | 降低base lr或增加warmup |
| 增加warmup | 更长的warmup期 |
| 数据清洗 | 移除异常样本 |
| BF16替代FP16 | 避免数值溢出 |
| 检查点回退 | 回退到spike前的检查点 |

**预防措施**：
```python
# 梯度监控
grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm)
if grad_norm > threshold:
    log_warning(f"Large gradient: {grad_norm}")

# 定期保存检查点
if step % save_interval == 0:
    save_checkpoint(model, optimizer, step)
```

<a id="问题9"></a>
### 问题9：流水线并行的Pipeline Bubble是什么？如何减少？

**参考答案**：

**Pipeline Bubble**：
```
流水线中GPU空闲等待的时间

朴素流水线（4 GPU，1 micro-batch）：
时间 →
GPU 0: F░░░░░░B░░░
GPU 1: ░F░░░░░░B░░
GPU 2: ░░F░░░░░░B░
GPU 3: ░░░F░░░░░░B

F=前向，B=反向，░=空闲（Bubble）
Bubble比例 ≈ (P-1)/P，P=流水线阶段数
```

**减少Bubble的方法**：

**1. Micro-batching**
```
将batch分成多个micro-batch

GPU 0: F₁F₂F₃F₄░░B₄B₃B₂B₁
GPU 1: ░F₁F₂F₃F₄░B₄B₃B₂B₁░
GPU 2: ░░F₁F₂F₃F₄B₄B₃B₂B₁░░
GPU 3: ░░░F₁F₂F₃F₄B₄B₃B₂B₁░░░

Bubble显著减少
```

**2. 1F1B调度**
```
交错前向和反向，减少激活值内存

GPU 0: F₁F₂F₃F₄B₁B₂B₃B₄
每完成一个micro-batch的前向，就开始其反向
```

**3. Interleaved Pipeline**
```
每个GPU负责多个非连续的层
进一步减少Bubble
```

<a id="问题10"></a>
### 问题10：张量并行是如何切分Attention层和FFN层的？

**参考答案**：

**FFN层切分**：

```
FFN(x) = GELU(xW₁)W₂

列切分W₁，行切分W₂：

     GPU 0          GPU 1
       │              │
x ────→ W₁ᵃ ──GELU──→ W₂ᵃ ──┐
       │              │      │ AllReduce
x ────→ W₁ᵇ ──GELU──→ W₂ᵇ ──┘
                             │
                             ↓
                             Y
```

**Attention层切分**：

```
多头注意力天然适合切分：

8头 + 2 GPU：
GPU 0: Q₀₋₃, K₀₋₃, V₀₋₃ → 头0,1,2,3
GPU 1: Q₄₋₇, K₄₋₇, V₄₋₇ → 头4,5,6,7

每个GPU独立计算自己的头
最后AllReduce合并输出
```

**通信位置**：
```
每个Transformer层需要2次AllReduce：
1. Attention输出后
2. FFN输出后
```

<a id="问题11"></a>
### 问题11：DDP和FSDP有什么区别？

**参考答案**：

**DDP（Distributed Data Parallel）**：
```
每个GPU持有完整模型
数据分片，梯度AllReduce同步

内存：每卡需要装下完整模型
通信：只同步梯度
```

**FSDP（Fully Sharded Data Parallel）**：
```
模型参数、梯度、优化器状态都分片
前向时收集参数，反向时分发梯度

内存：每卡只存1/N的模型状态
通信：需要收集参数和分发梯度
```

**对比表格**：

| 特性 | DDP | FSDP |
|-----|-----|------|
| 参数存储 | 完整 | 分片（1/N） |
| 内存效率 | 低 | 高 |
| 通信量 | 梯度AllReduce | 参数AllGather + 梯度ReduceScatter |
| 实现复杂度 | 简单 | 复杂 |
| 适用场景 | 模型装得下 | 模型装不下 |

**FSDP ≈ ZeRO-3的PyTorch实现**

<a id="问题12"></a>
### 问题12：大模型预训练的数据如何处理和配比？

**参考答案**：

**数据处理流程**：

```
原始数据
    ↓
语言检测（过滤非目标语言）
    ↓
去重（文档级、句子级）
    ↓
质量过滤（困惑度、长度、特殊字符比例）
    ↓
敏感信息处理（PII移除）
    ↓
Tokenization
    ↓
打包（拼接短文本，填充长文本）
    ↓
训练数据
```

**数据配比原则**：

| 原则 | 说明 |
|-----|------|
| 质量优先 | 高质量数据可多次采样 |
| 多样性 | 覆盖不同领域和任务 |
| 平衡性 | 避免某类数据主导 |

**典型配比**：
```
网页数据：60-70%（量大，但噪声多）
书籍：5-10%（高质量，采样2-3次）
代码：5-10%（编程能力）
论文：2-5%（专业知识）
百科：3-5%（知识密集，采样2-3次）
对话：2-5%（对话能力）
```

<a id="问题13"></a>
### 问题13：什么是梯度累积？什么场景需要使用？

**参考答案**：

**梯度累积原理**：
```
多次小batch前向反向，累积梯度后统一更新

等效batch_size = micro_batch × accumulation_steps × num_gpus
```

**代码示例**：
```python
accumulation_steps = 4
optimizer.zero_grad()

for i, batch in enumerate(dataloader):
    loss = model(batch) / accumulation_steps
    loss.backward()  # 梯度累积

    if (i + 1) % accumulation_steps == 0:
        optimizer.step()
        optimizer.zero_grad()
```

**使用场景**：

| 场景 | 说明 |
|-----|------|
| GPU内存不足 | 无法使用期望的batch size |
| 稳定训练 | 大batch通常更稳定 |
| 模拟大batch | 用小batch模拟大batch效果 |
| 单卡训练 | 没有多卡数据并行时 |

**注意事项**：
- Loss要除以accumulation_steps
- BN层可能受影响（每个micro-batch统计量不同）
- 等效batch size要与学习率匹配

<a id="问题14"></a>
### 问题14：学习率Warmup的作用是什么？

**参考答案**：

**Warmup原理**：
```
训练初期使用小学习率，逐渐增大到目标学习率

lr_t = base_lr × (t / warmup_steps), t < warmup_steps
```

**作用**：

| 作用 | 解释 |
|-----|------|
| 稳定初期训练 | 初期梯度估计不准，大lr容易发散 |
| 避免早期过拟合 | 防止模型快速收敛到局部最优 |
| Adam等自适应优化器需要 | 二阶矩估计需要积累 |
| 配合LayerNorm | 初期scale参数敏感 |

**典型配置**：
```
大模型训练：
- warmup_steps = 总步数的1-5%
- 或固定warmup_tokens（如375M tokens）

示例：
- 总训练：1T tokens
- warmup：3.75B tokens（约0.375%）
```

**学习率调度完整策略**：
```
Warmup → Peak LR → Cosine/Linear Decay → Min LR
```

<a id="问题15"></a>
### 问题15：对比DeepSpeed和Megatron-LM的特点

**参考答案**：

| 特性 | DeepSpeed | Megatron-LM |
|-----|-----------|-------------|
| 开发者 | Microsoft | NVIDIA |
| 主要特点 | ZeRO优化器 | 高效张量并行 |
| 易用性 | 更简单，配置驱动 | 较复杂，需要代码修改 |
| 并行支持 | ZeRO + Pipeline | 3D并行 |
| 性能优化 | 通用优化 | 针对NVIDIA GPU深度优化 |
| 社区 | 活跃，文档丰富 | NVIDIA官方支持 |

**DeepSpeed优势**：
- 零代码修改即可使用ZeRO
- 支持CPU/NVMe offload
- 与HuggingFace集成好
- 更新频繁

**Megatron-LM优势**：
- 张量并行效率更高
- NVIDIA GPU优化更深
- 大规模训练经验丰富
- 与NeMo框架集成

**实践建议**：
```
中等规模（10B以下）：DeepSpeed更方便
大规模训练（100B+）：Megatron-LM性能更好
最佳实践：Megatron-DeepSpeed（结合两者优势）
```
