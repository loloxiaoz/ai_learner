<a id="问题1"></a>
### 问题1：什么是RAG？为什么需要RAG？

**参考答案**：

**RAG定义**：
```
Retrieval-Augmented Generation（检索增强生成）
让LLM在生成答案时参考外部知识库
```

**为什么需要RAG**：

| 问题 | RAG如何解决 |
|-----|------------|
| 知识过时 | 知识库可实时更新 |
| 幻觉 | 答案有据可查，可追溯 |
| 领域知识缺乏 | 接入专业知识库 |
| 上下文长度限制 | 检索最相关部分 |

**RAG优势**：
```
1. 知识实时更新
   - 无需重新训练模型
   - 只需更新知识库

2. 可解释性强
   - 答案可以引用来源
   - 便于审计和纠错

3. 成本低
   - 不需要训练大模型
   - 可用较小模型+好的检索

4. 减少幻觉
   - 基于真实文档生成
   - 不鼓励编造
```

<a id="问题2"></a>
### 问题2：RAG和Fine-tuning有什么区别？如何选择？

**参考答案**：

**对比**：

| 维度 | RAG | Fine-tuning |
|-----|-----|-------------|
| 知识更新 | 实时，改知识库 | 需要重新训练 |
| 成本 | 低 | 高（需GPU） |
| 幻觉 | 减少（有出处） | 可能增加 |
| 可解释性 | 高 | 低 |
| 适用场景 | 知识密集型 | 风格/行为调整 |

**选择指南**：

```
选择RAG：
- 知识需要频繁更新
- 需要引用来源
- 领域知识丰富但结构化
- 成本敏感

选择Fine-tuning：
- 需要特定输出风格
- 需要学习复杂格式
- 知识相对稳定
- 对延迟敏感（RAG有检索开销）

组合使用：
- Fine-tuning学习任务格式和风格
- RAG提供实时知识
```

### 问题3：RAG系统的完整流程是什么？

**参考答案**：

**离线索引阶段**：
```
1. 文档加载
   - 支持PDF、Word、Markdown等格式
   - 提取文本内容

2. 文档切分
   - 按chunk_size切分
   - 保留overlap避免断裂

3. 向量化
   - Embedding模型编码
   - 生成向量表示

4. 存储索引
   - 存入向量数据库
   - 建立索引加速检索
```

**在线查询阶段**：
```
1. Query处理
   - 可选：Query改写/扩展
   - Query向量化

2. 相似检索
   - 向量数据库检索
   - 返回top-k相关文档

3. 重排序（可选）
   - Cross-Encoder精排
   - 提高相关性

4. 生成答案
   - 构造prompt：问题+检索结果
   - LLM生成答案
```

### 问题4：文档切分的chunk size应该如何选择？

**参考答案**：

**影响因素**：

| 因素 | 小chunk | 大chunk |
|-----|---------|---------|
| 检索精度 | 高 | 低 |
| 上下文 | 少 | 多 |
| 噪声 | 少 | 多 |
| 边界问题 | 可能断裂 | 较完整 |

**推荐值**：
```
通用场景：500-1000字符
技术文档：800-1500字符
对话数据：200-500字符
代码：按函数/类切分

overlap：chunk_size的10-20%
```

**选择方法**：
```python
# 实验验证最佳chunk_size
def evaluate_chunk_size(docs, test_queries, chunk_sizes):
    results = {}
    for size in chunk_sizes:
        chunks = split_documents(docs, chunk_size=size)
        vectorstore = create_vectorstore(chunks)

        # 计算检索召回率
        recall = evaluate_retrieval(vectorstore, test_queries)
        results[size] = recall

    return results
```

**参考答案**：

**Embedding定义**：
```
将文本映射到高维向量空间的过程
使得语义相似的文本具有相似的向量表示
```

**主流模型对比**：

| 模型 | 维度 | 语言 | 特点 |
|-----|------|------|------|
| OpenAI text-embedding-3 | 1536/3072 | 多语言 | API简单 |

| E5-large | 1024 | 英文 | 效果好 |
| GTE-large | 1024 | 中文 | 阿里开源 |
| Cohere embed | 1024 | 多语言 | 商业API |

**选择依据**：
```
1. 语言匹配
   - 中文：BGE-M3, GTE
   - 英文：E5, OpenAI
   - 多语言：BGE-M3

2. 性能vs成本
   - 高性能：OpenAI, Cohere
   - 本地部署：BGE, E5

3. 维度选择
   - 高维：精度高，存储大
   - 低维：速度快，存储小

4. Benchmark表现
   - MTEB排行榜
```

<a id="问题6"></a>
### 问题6：Dense Retrieval和Sparse Retrieval有什么区别？

**参考答案**：

**对比**：

| 特性 | Dense | Sparse |
|-----|-------|--------|
| 原理 | 向量相似度 | 关键词匹配 |
| 代表 | 向量检索 | BM25 |
| 语义理解 | 强 | 弱 |
| 精确匹配 | 弱 | 强 |
| 计算开销 | 高 | 低 |

**Dense Retrieval**：
```
优点：
- 理解语义相似性
- "狗"可以匹配"犬"

缺点：
- 可能忽略关键词
- 计算成本高
```

**Sparse Retrieval（BM25）**：
```
优点：
- 关键词精确匹配
- 计算效率高
- 可解释性好

缺点：
- 无法理解语义
- "狗"无法匹配"犬"
```

**选择建议**：
```
精确查询（专有名词、代码）→ Sparse
语义查询（自然问题）→ Dense
综合场景 → Hybrid
```

<a id="问题7"></a>
### 问题7：什么是混合检索（Hybrid Search）？有什么好处？

**参考答案**：

**定义**：
```
结合Dense和Sparse检索的方法
综合语义相似性和关键词匹配
```

**实现方式**：
```python
def hybrid_search(query, k=10, alpha=0.5):
    # Dense检索
    dense_results = dense_retriever.search(query, k=k*2)

    # Sparse检索（BM25）
    sparse_results = bm25_retriever.search(query, k=k*2)

    # 分数融合
    combined = {}
    for doc, score in dense_results:
        combined[doc.id] = alpha * normalize(score)
    for doc, score in sparse_results:
        if doc.id in combined:
            combined[doc.id] += (1-alpha) * normalize(score)
        else:
            combined[doc.id] = (1-alpha) * normalize(score)

    # 排序返回
    sorted_docs = sorted(combined.items(), key=lambda x: x[1], reverse=True)
    return sorted_docs[:k]
```

**好处**：
```
1. 兼顾语义和关键词
2. 减少漏检
3. 更鲁棒

研究表明：
混合检索通常比单一方法提升5-15%
```

<a id="问题8"></a>
### 问题8：什么是重排序（Reranking）？为什么需要它？

**参考答案**：

**定义**：
```
对初步检索结果进行精细排序
使用更强大的模型（Cross-Encoder）重新评估相关性
```

**为什么需要**：
```
初步检索：
- Bi-Encoder：独立编码query和doc
- 速度快但精度有限

重排序：
- Cross-Encoder：query和doc一起编码
- 精度高但速度慢

两阶段策略：
1. 粗排：快速检索top-100
2. 精排：精细重排得到top-10
```

**实现**：
```python
from sentence_transformers import CrossEncoder

reranker = CrossEncoder('BAAI/bge-reranker-large')

def rerank(query, documents, top_k=5):
    pairs = [[query, doc.page_content] for doc in documents]
    scores = reranker.predict(pairs)

    # 按分数排序
    doc_scores = list(zip(documents, scores))
    doc_scores.sort(key=lambda x: x[1], reverse=True)

    return [doc for doc, score in doc_scores[:top_k]]
```

**效果提升**：
```
通常可以提升5-20%的检索精度
对于复杂查询效果更明显
```

<a id="问题9"></a>
### 问题9：什么是HyDE？原理是什么？

**参考答案**：

**HyDE定义**：
```
Hypothetical Document Embeddings
假设性文档嵌入
```

**原理**：
```
问题：用户query和文档的向量空间可能不一致
- Query："什么是Transformer？"
- 文档："Transformer是一种神经网络架构..."

HyDE方案：
1. 用LLM为query生成假设性答案
2. 用假设答案的向量去检索
3. 假设答案和真实文档更相似
```

**流程**：
```python
def hyde_search(query, llm, retriever):
    # Step 1: 生成假设性文档
    prompt = f"请回答以下问题（可以假设性回答）：{query}"
    hypothetical_doc = llm(prompt)

    # Step 2: 用假设文档检索
    results = retriever.search(hypothetical_doc)

    return results
```

**适用场景**：
```
优点：
- 提高query和doc的匹配度
- 对开放性问题效果好

缺点：
- 增加LLM调用开销
- 假设答案可能误导检索

适用：
- 复杂问题
- query和doc表达方式差异大
```

<a id="问题10"></a>
### 问题10：如何评估RAG系统的效果？

**参考答案**：

**评估维度**：

| 维度 | 指标 | 评估方法 |
|-----|------|---------|
| 检索质量 | Recall@K, MRR | 标注测试集 |
| 生成质量 | 正确性、相关性 | 人工评估/LLM评估 |
| 端到端 | 答案准确率 | QA测试集 |

**检索评估**：
```python
def evaluate_retrieval(retriever, test_queries, ground_truth):
    metrics = {"recall@5": [], "mrr": []}

    for query, relevant_docs in zip(test_queries, ground_truth):
        retrieved = retriever.search(query, k=10)
        retrieved_ids = [doc.id for doc in retrieved]

        # Recall@5
        recall = len(set(retrieved_ids[:5]) & set(relevant_docs)) / len(relevant_docs)
        metrics["recall@5"].append(recall)

        # MRR
        for i, doc_id in enumerate(retrieved_ids):
            if doc_id in relevant_docs:
                metrics["mrr"].append(1 / (i + 1))
                break
        else:
            metrics["mrr"].append(0)

    return {k: np.mean(v) for k, v in metrics.items()}
```

**生成评估**：
```python
def evaluate_generation(rag_system, test_cases):
    """使用LLM评估答案质量"""
    scores = []
    for question, ground_truth in test_cases:
        answer = rag_system.query(question)

        eval_prompt = f"""
        问题：{question}
        标准答案：{ground_truth}
        模型答案：{answer}

        请评估模型答案的质量（1-5分）：
        """
        score = llm_judge(eval_prompt)
        scores.append(score)

    return np.mean(scores)
```

### 问题11：什么是Agent？与RAG有什么区别？

**参考答案**：

**Agent定义**：
```
能够感知环境、做出决策、执行行动的智能体
具备推理、规划、工具使用能力
```

**核心能力**：
```
1. 推理规划：分析任务，制定计划
2. 工具使用：调用外部API和工具
3. 记忆：保持上下文和历史
4. 反思：评估结果，调整策略
```

**与RAG的区别**：

| 方面 | RAG | Agent |
|-----|-----|-------|
| 能力 | 检索+生成 | 推理+多工具 |
| 工具 | 1个（检索器） | 多个 |
| 交互 | 通常单轮 | 多轮迭代 |
| 自主性 | 低 | 高 |
| 复杂度 | 低 | 高 |

**关系**：
```
Agent可以包含RAG作为其中一个工具
Agent = LLM + Tools（RAG是工具之一）

示例Agent工具：
- 搜索引擎
- RAG知识库
- 计算器
- 代码执行器
- API调用
```

<a id="问题12"></a>
### 问题12：什么是Function Calling？如何实现？

**参考答案**：

**定义**：
```
让LLM决定调用哪个函数，并生成函数参数
本质是LLM的结构化输出能力
```

**工作流程**：
```
1. 用户输入 + 函数定义 → LLM
2. LLM判断是否需要调用函数
3. 如果需要，输出函数名和参数（JSON格式）
4. 应用程序执行函数
5. 函数结果返回给LLM
6. LLM生成最终回答
```

**实现示例**：
```python
import openai
import json

# 定义函数
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "获取天气信息",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "城市名"},
                    "date": {"type": "string", "description": "日期"}
                },
                "required": ["city"]
            }
        }
    }
]

# 调用
response = openai.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "北京明天天气如何？"}],
    tools=tools
)

# 处理
if response.choices[0].message.tool_calls:
    call = response.choices[0].message.tool_calls[0]
    func_name = call.function.name
    args = json.loads(call.function.arguments)
    # args = {"city": "北京", "date": "明天"}

    # 执行函数
    result = globals()[func_name](**args)
```

<a id="问题13"></a>
### 问题13：什么是ReAct模式？请举例说明

**参考答案**：

**定义**：
```
ReAct = Reasoning + Acting
推理和行动交替进行的Agent模式
```

**流程**：
```
Thought（思考）→ Action（行动）→ Observation（观察）→ 循环
直到得出最终答案
```

**示例**：
```
问题：2023年诺贝尔物理学奖得主的年龄是多少？

Thought: 我需要先查询2023年诺贝尔物理学奖得主是谁
Action: search("2023年诺贝尔物理学奖得主")
Observation: 2023年诺贝尔物理学奖授予Pierre Agostini、Ferenc Krausz和Anne L'Huillier

Thought: 有三位得主，我需要查询每位的出生年份
Action: search("Pierre Agostini 出生年份")
Observation: Pierre Agostini生于1941年

Action: search("Ferenc Krausz 出生年份")
Observation: Ferenc Krausz生于1962年

Action: search("Anne L'Huillier 出生年份")
Observation: Anne L'Huillier生于1958年

Thought: 现在我可以计算年龄了
Action: calculate("2023 - 1941, 2023 - 1962, 2023 - 1958")
Observation: [82, 61, 65]

Answer: 2023年诺贝尔物理学奖三位得主的年龄分别是：
Pierre Agostini 82岁，Ferenc Krausz 61岁，Anne L'Huillier 65岁
```

<a id="问题14"></a>
### 问题14：RAG系统中常见的问题和解决方案？

**参考答案**：

**问题1：检索不相关**
```
原因：
- Embedding模型不匹配
- chunk切分不合理
- Query表达与文档差异大

解决：
- 选择更好的Embedding模型
- 调整chunk size和overlap
- Query改写/扩展
- 使用HyDE
```

**问题2：上下文不够**
```
原因：
- chunk太小
- 检索数量不足

解决：
- 增大chunk size
- 增加top-k
- 使用父文档检索（检索chunk但返回整个文档）
```

**问题3：检索到但答案不对**
```
原因：
- Prompt设计问题
- LLM能力不足
- 文档质量差

解决：
- 优化prompt，强调基于文档回答
- 使用更强的LLM
- 清洗文档数据
```

**问题4：响应延迟高**
```
原因：
- 检索慢
- LLM生成慢

解决：
- 优化向量索引（IVF, HNSW）
- 使用更快的Embedding模型
- 考虑缓存常见查询
- 流式输出
```
